# Frontend UI & Reactivity (RRG)

> **Контекст**: Правила организации UI слоя и управления состоянием (Reactive Reliability Gate).
> **Scope**: `app/`, `shared/components/`

## 1. Запрет хардкода текстов (Hardcode Ban)
Все пользовательские тексты (кнопки, заголовки, тултипы) должны браться из единых конфигураций (SSOT).
- Запрещено использовать хардкод строк при прямом назначении `innerText`, `textContent`, `title`, `ariaLabel` в DOM-элементах.
- За соблюдением следит AST-линтер `is/scripts/tests/lint-frontend-hardcode-ast.test.js`.

## 2. Zod-валидация UI-конфигов (Type Safe Design System)
Конфигурации интерфейса (`tooltips-config.js`, `modals-config.js`) валидируются с помощью Zod-схем.
- Схемы лежат в `core/contracts/ui-contracts.js`.
- Проверка идет в `validate-frontend-ui-configs.test.js`.
- **@causality**: Опечатка в конфиге (например, отсутствие обязательного ключа) не должна приводить к "тихим" ошибкам на клиенте. Fail-fast на этапе тестирования.

## 3. Reactive Reliability Gate (RRG)
RRG — это концепция фундаментальной надежности реактивного слоя.
1. **Единый источник состояния**: Vue реактивность (или `appStore`) должна быть четко структурирована.
2. **Запрет мутаций вне контрактов**: Избегать неконтролируемого расползания состояния (например, ручной синхронизации двух независимых переменных вместо `computed`).
3. Для критичных компонентов (таблицы, списки) проверяется соблюдение контрактов через автоматические тесты.

## 4. Использование Vue "No-Build"
Проект использует Vue 3 через глобальный тег `<script>` без бандлеров (Vite/Webpack), чтобы оставаться переносимым и работать по протоколу `file://`.
- Шаблоны выносятся в отдельные JS-файлы как строки (например, `app/templates/`).
- Загрузка модулей идет строго в нужном порядке через `core/module-loader.js`.
