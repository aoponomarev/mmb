# План тестирования и проверок (Testing Strategy)

> Категория: План тестовой стратегии миграции
> Статус: В работе
> Источник: План_Testing_Strategy.md из Donor Architecture

---

## 1. Контекст и границы

Контур определяет, как подтверждать корректность миграции на каждом этапе:
unit / integration / smoke / preflight / regression. Мы используем встроенный `node:test` (см. `is/skills/arch-foundation.md`).

## 2. Цели / Non-goals

**Цели:**
- ввести предсказуемый quality-gate pipeline через `node:test`;
- проверить критичные потоки до merge и после deploy;
- привязать тесты к этапам миграции.

**Non-goals:**
- 100% покрытие тестами всего legacy-кода;
- внедрение тяжелых e2e фреймворков (Playwright/Cypress) на текущем этапе.

## 3. Почему этот подход

- миграция без поэтапных проверок дает отложенные системные ошибки;
- preflight (Zod-контракты) + `node:test` (unit/integration) дают наилучшее соотношение цена/надежность.

## 4. Альтернативы

- только ручная проверка — отклонено, слишком высокий риск регрессий.
- использование Jest / Vitest — отклонено, так как `node:test` закрывает 99% потребностей без лишних зависимостей.

## 5. Зависимости

- Успешная миграция слоя Backend Core (API Providers, Market Snapshot).

## 6. Риски и снижение

- flaky tests -> детерминированные моки (как в `backend-market-runtime.test.js`).
- ложное чувство безопасности -> обязательные проверки валидации контрактов (Zod) перед вызовом логики.
