# Академический словарь терминов архитектуры (Glossary)

> Этот словарь является SSOT (Единым Источником Правды) для терминологии проекта. ИИ-агенты и разработчики обязаны опираться на эти определения при написании документации, комментариев и обсуждении архитектуры.
> 
> **Правило ведения глоссария:** Разделы и термины добавляются свободно, без нумерации. При добавлении новых секций и терминов необходимо соблюдать строгий порядок следования — от наиболее общих/глобальных архитектурных понятий (макро-уровень) к более частным и атомарным (микро-уровень).

## Топология и Структура

- **Слой (Layer)** — Уровень вертикальной иерархии архитектуры. Слой предоставляет услуги тем, кто выше, и использует ресурсы тех, кто ниже. *Пример:* `app/` (Presentation Layer), `core/` (Business Logic Layer), `is/` (Infrastructure Layer). **Не путать с Контуром.**
- **Контур (Contour / Perimeter)** — Замкнутая система с четко очерченной границей, имеющая свой цикл (вход-обработка-выход) или периметр безопасности. *Пример:* Ingest-контур (регулярный сбор данных из API), RRG-контур (периметр правил надежности фронтенда). **Не путать со Слоем** — контур предполагает замкнутый цикл потока данных, а не вертикальную иерархию.
- **Плоскость (Plane)** — Логическое пространство с единой ответственностью, обычно противопоставляемое другой плоскости. *Пример:* Control Plane (управление: MCP, preflight, causality) vs Data Plane (данные: провайдеры, кэш, БД).
- **Домен (Domain / Bounded Context)** — Область знаний или ответственности с собственным языком и правилами. *Пример:* Домен портфолио (`portfolio-engine`), домен рыночных данных (`market-snapshot`).
- **Сегмент (Segment)** — Горизонтальное логическое разделение внутри одного слоя. *Пример:* Сегмент Auth или Сегмент Portfolios внутри слоя `app/`.
- **Граница (Boundary)** — Четкий предел ответственности между доменами, слоями или модулями. Взаимодействие через границу допускается только через контракты.
- **Пространство имён (Namespace)** — Логическое разграничение имён и видимости для предотвращения коллизий. *Пример:* `window.cmpButton` vs `window.appHeader` — разные пространства по конвенции префиксов.

## Исполняемые Единицы

- **Модуль (Module)** — Физическая единица упаковки кода (конкретный файл или папка с `index.js`), инкапсулирующая логику.
- **Сервис (Service)** — Функциональная единица бизнес-логики, выполняющая конкретную задачу (часто оркеструет другие модули). *Пример:* `portfolio-engine`, `market-snapshot-service`. **Не путать с Провайдером** — сервис содержит бизнес-логику, провайдер лишь добывает/адаптирует данные.
- **Провайдер (Provider) / Адаптер (Adapter)** — Модуль, который инкапсулирует работу с внешним миром (API, БД) и адаптирует чужие форматы к внутренним контрактам проекта. *Пример:* `YandexCacheProvider`. **Не путать с Сервисом.**
- **Фасад (Facade)** — Упрощенный интерфейс к сложной подсистеме. *Пример:* `DataProviderManager`, который скрывает за собой логику выбора нужного провайдера.
- **Оркестратор (Orchestrator)** — Сервис, который управляет порядком вызова других сервисов/провайдеров. **Не путать с Хореографом** — оркестратор управляет централизованно, хореограф работает через события без центра.

## Потоки и Коммуникация

- **Конвейер (Pipeline)** — Последовательная цепочка обработки данных, где выход одного этапа становится входом следующего. *Пример:* Data Pipeline (CoinGecko API -> PostgreSQL -> UI).
- **Транспорт (Transport)** — Способ/канал передачи данных между компонентами. *Пример:* HTTP, WebSocket, gRPC, postMessage.
- **Шина (Bus)** — Общая среда для передачи событий между несвязанными компонентами. *Пример:* Event Bus, Message Bus.
- **Перехватчик (Middleware / Interceptor)** — Код, выполняющийся в разрыве между вызовом и исполнением для добавления сквозной функциональности. *Пример:* CORS-перехватчик в Cloudflare Worker, логирование запросов.

## Правила и Связи

- **Контракт (Contract)** — Строгое соглашение о структуре данных или интерфейсе взаимодействия. Приложение обязано проверять данные на соответствие контракту. *Пример:* Zod-схема для Domain-сущности, JSON-реестр.
- **Гейт (Gate)** — Точка жесткого контроля (часто скрипт в CI или `preflight`), которая проверяет соблюдение контрактов или политик. Гейт блокирует выполнение (`exit(1)`) при нарушении. **Не путать с Модулем или Сервисом** — гейт не несёт бизнес-логики, он только проверяет.
- **Политика (Policy)** — Набор правил и ограничений, которые описывают, как должна вести себя система или разработчик (могут проверяться гейтами).
- **Мост (Bridge)** — Компонент, связывающий две принципиально разные среды выполнения. *Пример:* MCP-сервер связывает файловую систему проекта с ИИ-агентом.
- **Схема (Schema)** — Формальное описание структуры данных, используемое для валидации. *Пример:* Zod-схема, JSON Schema.

## Состояние и Процессы

- **Жизненный цикл (Lifecycle)** — Последовательность состояний или этапов, которые проходит сущность (артефакт, компонент, данные) от момента создания до уничтожения или архивации. *Пример:* Lifecycle AIS-спецификации: `draft` -> `incomplete` -> `complete`.
- **Реестр (Registry)** — Централизованный, доверенный список ключей или путей. *Пример:* `causality-registry.md`, `id-registry.json`, `code-file-registry.json`.
- **Снимок (Snapshot)** — Зафиксированное состояние системы или данных в конкретный момент времени. *Пример:* `coin_market_cache` — текущий снимок рыночных данных.
- **Кэш (Cache)** — Временное хранилище для ускорения повторного доступа к данным. **Не путать с Реестром** — кэш содержит копию с ограниченным временем жизни (TTL), реестр — каноничный первоисточник.
- **Якорь (Anchor)** — Точка привязки кода к смыслу или документации. *Пример:* `@causality` и `@skill` в заголовках кодовых файлов.
- **Казуальность (Causality)** — Зафиксированная причинно-следственная связь между архитектурным решением и его обоснованием. *Пример:* `#for-anti-calque` — хеш-обоснование запрета транслитерации.

## Domain Terms (Market Data)

- **Стейблкоин (Stablecoin / Pegged Asset)** — Токен, привязанный к базовому активу: фиат (USD, EUR), металлы (золото, серебро, платина, палладий), сырьё (нефть) и т.п. В проекте: `coins.json` (GitHub CDN) хранит реестр в виде `stable.fiat.{peg}` и `stable.commodity.{peg}`; генератор опирается на market-cache + stablecoin-filter (по цене), а metal/raw split внутри `commodity` вычисляется программно по peg.
