# API Layer & Providers (Слой данных и провайдеров)

> **Ключ навыка:** `core/skills/api-layer`

Этот скил описывает контракты и принципы работы слоя API (бэкенд) в нашем приложении.

## Основные принципы (Active Causality & Policies)

1. **Fail-Fast (Быстрый отказ):**
   Провайдеры данных (например, `CoinGeckoProvider`, `BinanceMetricsProvider`) не должны иметь скрытых цепочек фоллбеков или ретраев до бесконечности. Внешние ошибки (особенно таймауты и HTTP 500/400/429) должны быть возвращены наверх в виде стандартизированной ошибки `BackendCoreError`.
   - `@causality`: Скрытие ошибок в провайдере усложняет отладку и может привести к зависанию оркестратора. Провайдер должен явно отваливаться при исчерпании лимитов.

2. **Best-Effort Orchestration (Оркестрация без блокировок):**
   Сервисы, агрегирующие данные из нескольких независимых источников (например, `MarketMetricsService`), обязаны использовать `Promise.allSettled` для методов массовой загрузки (вроде `getAllBestEffort`).
   - `@causality`: Отказ одного провайдера (например, Yahoo Finance заблокировал IP для VIX) не должен валить весь ответ для клиента. Здоровые метрики должны возвращаться, а проблемные — заменяться структурированной ошибкой (например, `{ source: "error", error: { ... } }`).

3. **Rate Limiting & Journaling:**
   Все обращения к внешним API проходят через `RequestRegistry` или аналогичный rate-limiter, чтобы не получать баны (HTTP 429).
   - При получении HTTP 429 интервал запросов увеличивается (multiplier), а при успехе сбрасывается.
   - `@causality`: Бесплатные API имеют жесткие лимиты, поэтому мы должны проактивно ждать, чтобы избегать постоянных ошибок.

4. **Кэширование на файловой системе (MMB-native):**
   Вместо сложного хранения в памяти или localStorage на сервере, используется легковесный детерминированный `DataCacheManager`, опирающийся на Node.js `fs`.
   - `@causality`: Файловый кэш выживает при перезапусках скрипта, легко очищается и не забивает оперативную память приложения.

5. **Изоляция контрактов:**
   Форматы входных данных строго проверяются (через Zod-контракты или утилиты валидации), прежде чем запрос уйдет в сеть.

6. **Разделение слоев (Transport, HTTP, Service):**
   - **Service (Домен)**: Оркестрирует получение данных. Ничего не знает об HTTP-запросах.
   - **Transport**: Адаптер, который ловит ошибки домена (например, `BackendCoreError`) и преобразует их в структуры `{ status, body }`. `@causality`: Изолирует бизнес-логику от специфики протокола.
   - **HTTP Handler**: Фреймворк-независимый обработчик (не привязан к Express/Koa/Hono), который разбирает URL, обрабатывает CORS (`OPTIONS`), Request ID и заголовки, затем передает чистый вызов в Transport. `@causality`: Стандартизация ответов и заголовков (например, `x-request-id`) в одном месте.

7. **Composition Root (Сборка зависимостей):**
   Все зависимости (Кэш, Сервисы, Провайдеры, Роутеры) инициализируются в одном месте (`backend-market-runtime.js`).
   - `@causality`: Отказ от синглтонов (Singleton) позволяет легко мокать зависимости в e2e тестах и изолировать инстансы (например, для разных тенантов).
